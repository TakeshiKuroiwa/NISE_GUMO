# 行動規範（Code of Conduct）

## はじめに

NISE_GUMO は、プロジェクト管理を少し楽にするためのモックアップです。
このリポジトリに参加する人は、コードを書く人、Issue を立てる人、ドキュメントの誤字を見つける人、未来の自分を救う人など、全員が大切なコントリビューターです。

技術的な議論は歓迎します。机を叩くより、再現手順を書いたほうが早く直ります。

---

## 期待するふるまい

- 相手を尊重し、攻撃ではなく改善に向けて発言する
- バグ報告や提案では、背景、再現手順、期待する動作をできるだけ具体的に書く
- レビューでは、問題点と理由をセットで伝える
- 意見が分かれた場合は、目的、制約、ユーザーへの影響に立ち戻る
- ドキュメントやコメントを通じて、未来の参加者が迷子にならないようにする

---

## 歓迎するコミュニケーション

- 「この実装意図を教えてください」
- 「このケースでは別案もありそうです」
- 「再現手順を追加しました」
- 「ここはモック段階なので、いったん軽い実装にしています」
- 「昨日の自分が読める程度にコメントを足しました」

軽いユーモアは歓迎します。ただし、誰かを笑うのではなく、作業の緊張を少しほどくために使ってください。

---

## 避けてほしいコミュニケーション

- 人格、属性、経験年数、所属などを攻撃する発言
- 「明らかに間違い」だけで理由や代替案を示さないレビュー
- 必要な文脈を共有せずに相手を急かすこと
- Issue や PR の場で無関係な議論を長く続けること
- 絵文字や冗談を、相手が困るほど連打すること

---

## Pull Request と Issue の基本

### Issue

良い Issue には、次の情報があると助かります。

1. 何に困っているか
2. どの画面、機能、ファイルに関係するか
3. 再現手順、または提案したい動作
4. 期待する結果と実際の結果
5. スクリーンショットや参考情報（あれば）

「なんか変です」は詩としては味がありますが、修正担当者には少し難問です。

### Pull Request

PR はなるべく小さく、目的をひとつに絞ってください。

- 変更内容と理由を書く
- 関連 Issue があれば紐づける
- 必要に応じてスクリーンショットを添える
- ビルドや動作確認の結果を書く
- AIを活用した場合は、必要に応じて [AGENTS.md](./AGENTS.md) の観点で指示や成果物を確認する

## 禁止事項

以下の行為を禁止します。

| 行為 | 対応方針 |
|------|----------|
| ハラスメント、差別、脅迫 | リポジトリ管理者が確認し、必要に応じて参加制限を行います |
| 他者の個人情報の無断公開 | 直ちに削除・非公開化を検討します |
| 悪意あるコードや機密情報の混入 | 速やかに修正し、影響範囲を確認します |
| `master` への無断直接 push | 原則として PR ベースの運用へ戻します |
| レビューなしでの重要変更のマージ | 変更範囲に応じて追加確認を求めます |
| コピペのみで出典や意図が不明な貢献 | 内容確認や修正を依頼します |

---

## 執行について

問題が発生した場合は、リポジトリ管理者へ連絡してください。
報告には、発生場所、関係する Issue/PR、時系列、スクリーンショットなど、確認に必要な情報を含めてください。

報告者のプライバシーを尊重し、公正かつ迅速な対応を目指します。
「無視」は対応方針ではありません。少なくとも、見たことは見たと言える運用を目指します。

---

## 最後に

NISE_GUMO はまだモックアップ段階です。仕様も実装も、これから育てていく余白があります。
議論は具体的に、レビューは親切に、コミットメッセージは「fix」だけで済ませず、できれば何を直したかを書いてください。

この行動規範は [Contributor Covenant](https://www.contributor-covenant.org) v2.1 を参考にしつつ、NISE_GUMO の現在の開発段階に合わせて作成しています。
