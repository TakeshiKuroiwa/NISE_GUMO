# 行動規範（Code of Conduct）
# ――あるいは「コードの道（Tao of Code）」

> *「道の道とすべきは、常の道にあらず」― 老子*  
> *「つまり？」― 私たち全員*  
> *「まず README を読め」― 老子*

---

## 🌿 はじめに：そもそも「あなた」とは何か

このプロジェクトに参加するすべての人は、コントリビューター、メンテナー、Issue 荒らし予備軍を問わず、本質的に**道（Tao）の一部**です。

荘子が言ったように、蝶の夢を見た人間なのか、人間の夢を見た蝶なのかはわかりませんが、どちらにせよ PR のレビューはちゃんとしてください。夢の中でもコードの品質は大切です。

老子も言っています。*「千里の道も一歩から」* ――まず README を読むことから始めましょう。次に CONTRIBUTING.md を。その後で Issue を開いてください。順番が大事です。

---

## ☯️ 第一の道：無為（Wu Wei）のコミット

> *「為すことなくして、為されないことはない」― 老子*

良いコードとは、川が流れるように自然に書かれたコードです。  
**force push は川をせき止める行為です。** やめましょう。

- ✅ 流れに沿ってコードを書く（闘わず、諭さず、ただ書く）
- ✅ レビューの流れに逆らわず、抵抗せず、でも主張はする
- ❌ `git push --force` を master に対して行うことは、宇宙の調和を乱します
- ❌ 「俺のやり方が正しい」という執着は、荘子の言う「小知」の罠です
- ❌ コミットメッセージ「fix」だけは、老子が最も嫌う「名のつけられないもの」です

> *「大知は小知に如かず」― 荘子*  
> 大きな問題を小さなコミットに分けましょう。それが大知です。

---

## 🌊 第二の道：静（せい）の Issue

> *「重さは軽さの根、静けさは動きの主」― 老子*

Issue を開く前に、まず静かにしてください。5分間。  
本当に Issue が必要かどうか、自分の内側に問いかけてください。  
その5分が惜しければ、少なくとも検索はしてください。

**「今この瞬間に」存在する良い Issue の条件：**

1. 再現手順がある（「なんか壊れてます」は Issue ではなく詩です）
2. 期待する動作が書かれている
3. 怒りから書かれていない（一晩寝かせてから送信を。荘子も妻の死に涙した後、歌った）
4. タイトルが「バグ」だけではない（バグは虫、あなたの Issue は何の虫ですか）

> *「知者は弁ぜず、弁者は知らず」― 老子*  
> 長い Issue が必ずしも良い Issue ではありません。簡潔に。

## 🌀 第三の道：空（Emptiness）のプルリクエスト

荘子の料理人は牛を解体するとき、刃を骨と骨の間の**空間**に通しました。  
あなたの PR も同じように、既存のコードの隙間にそっと差し込んでください。  
3000行の差分は、骨ごと叩き割る斧です。メンテナーの心も割れます。

**PR の悟り度チェックリスト：**

- [ ] 1 PR = 1 つの目的（散漫な心は散漫なコードを生む）
- [ ] 変更行数は最小限（少ないほど道に近い）
- [ ] テストを書いた（無常に備えよ。明日のあなたは今日のあなたではない）
- [ ] セルフレビューした（自己観察は荘子の基本）
- [ ] PR の説明文がある（「変更しました」は説明ではなく事実の陳述です）

> *「曲がれば全きを保てる、曲がることで直くなる」― 老子*  
> PR が差し戻されても、それは道の一部です。泣かないでください（泣いてもいいです）。

## 🙏 第四の道：コミュニケーションの純粋性

老子はこう教えます。*「上善は水の若し」*  
水は争わず、しかし岩をも穿ちます。  
レビューコメントもそうあってください。たとえ相手の変数名が `x2` でも。

### 許容される表現

- 「このアプローチについて別の観点から考えてみませんか？🙏」
- 「なぜここでこのパターンを選んだか教えてもらえますか？」
- 「素晴らしい実装です。一点だけ気になる箇所があります」
- 「私も同じ間違いをしたことがあります（昨日）」
- 「荘子ならこう書くかもしれません…（代替案を添付）」

### 許容されない表現

- 「これは明らかに間違っている」（道において明らかなことは何もない）
- 「前任者が書いたコードが最悪」（前任者もまた道の一部です。コードもそうです）
- 「早くマージして」（急かすことは流れに逆らうことです）
- 絵文字の過剰使用（🙏🙏🙏🙏は一つの🙏より弱い。老子なら「多言数窮」と言う）

## 🌸 第五の道：執着からの解放（依存関係編）

> *「所有することなく、なすことなく、コントロールすることなく」― 老子*

npm パッケージへの執着は苦しみの源です。荘子も生前、`left-pad` 事件を知っていたら同じことを言ったでしょう（たぶん）。

- 必要最小限の依存を持つこと（あれもこれもインストールするのは「小知」の業です）
- バージョンへの固執を手放すこと、ただし `package-lock.json` はコミットすること
- `node_modules` を Git にコミットした者は、次の Issue で 108 回のレビューコメントを受けるでしょう
- 使われていない依存関係は、荘子の言う「無用の用」とは違います。ただの無用です

## 📖 第六の道：無知の知（ドキュメント編）

老子は言いました。*「知る者は語らず、語る者は知らず」*  

ただしソフトウェアにおいては、**知る者も語るべきです。** README に書いてください。道を知っているなら、他者にも道を示しましょう。

- コメントのないコードは、解説のない公案（コアン）です――一部の人には美しいが、99%の人には苦痛です
- 「自明である」と思ったことは、だいたい自明ではありません。荘子の料理人が刃の道を語ったように、あなたもコードの道を語ってください
- 未来の自分も他人です。親切にしてあげてください（彼はあなたのことを恨んでいます、今のところ）
- 「あとでドキュメントを書く」は「ついに川が上流に流れた」と同義です

## ⚡ 禁止事項（天道に逆らうもの）

以下の行為は、このコミュニティおよび自然の摂理により禁じられています：

| 行為 | 道家的解釈 |
|------|------|
| ハラスメント・差別 | エゴの最も粗い表れ。即座に対処します |
| 他者の個人情報の無断公開 | プライバシーは聖なる空間。老子も「名を秘めよ」と言う |
| 悪意あるコード（マルウェア等） | 害は巡り巡って己に返る（老子の「天網恢恢」） |
| `master` への直接 push | 天道に逆らう行為。川を逆流させるな |
| レビューなしでのマージ | 一人の悟りは悟りではない。荘子も「独善」を戒めた |
| ミーティングで話し続けること | *「多言数窮、中を守るに如かず」― 老子*。沈黙もコミュニケーションです |
| コピペのみのコントリビューション | 荘子の料理人は技を盗まず、道を学んだ |

## 🌙 執行について

問題が発生した場合はリポジトリ管理者へ。

すべての報告は、老子の水のごとく、**争わず、しかし確かに受け取られます。**  
「無為」は「無視」ではありません。老子もそこは区別しています（たぶん）。

対応フロー：

```
問題発生
   ↓
深呼吸（全員）―「重さは軽さの根、静けさは動きの主」
   ↓
報告を静かに受け取る
   ↓
公正かつ迅速に対応する（無為とは怠慢ではない）
   ↓
学びを得て、手放す―「為して争わず」
```

報告者のプライバシーは、水が岩を包むように守られます。
